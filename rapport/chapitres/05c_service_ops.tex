\section{Architecture API Gateway}

L'API Gateway expose le système multi-agents via REST, sans jamais exposer le graphe LangGraph directement à l'interface graphique. Implémentée avec FastAPI sur Uvicorn (\texttt{src/api.py}), elle suit un patron de découplage strict : l'interface NiceGUI ne dialogue qu'avec des points d'accès HTTP, jamais avec le graphe. Cette séparation produit quatre bénéfices.

D'abord, l'\textbf{indépendance des cycles de développement}. Les membres de l'équipe peuvent se répartir le \textit{back-end} (graphe et agents) et l'interface, sans dépendance circulaire entre les deux paquets Python.

Ensuite, la \textbf{testabilité} : \texttt{tests/test\_api.py} (environ 22 cas) utilise \texttt{TestClient} de FastAPI et des \textit{mocks} pour valider les réponses HTTP sans démarrer aucun agent. Chaque endpoint (\texttt{/recruter}, \texttt{/rapport}, \texttt{/hitl}, \texttt{/runs/\{id\}/stream}) est couvert en isolation.

Troisièmement, la \textbf{flexibilité de déploiement}. Le \textit{back-end} LLM peut basculer (Ollama vers OpenAI) ou la topologie d'agents évoluer sans régénération de la couche présentation. Le fichier \texttt{docker-compose.yml} déclare API et UI comme deux services indépendants : \texttt{sma-api} peut redémarrer sans déconnecter l'utilisateur côté UI au-delà du \textit{retry} SSE.

Enfin, l'\textbf{instrumentation décentralisée}. Les métriques et journaux du pipeline (voir section observabilité) sont collectés en mémoire de l'API, exportés en JSON dans \texttt{logs/metriques\_*.json} et consultables via \texttt{GET /metrics}, sans dépendance à une base externe.

Les endpoints principaux sont : \texttt{POST /recruter} (lance un \textit{run} en thread de fond via \texttt{BackgroundTasks}), \texttt{GET /rapport/\{run\_id\}} (état et résultats), \texttt{GET /runs/\{run\_id\}/stream} (flux SSE temps réel) et \texttt{POST /runs/\{run\_id\}/hitl} (validation humaine). Chaque \textit{run} est identifié par un UUID v4 et stocké dans un dictionnaire en mémoire, suffisant pour le démonstrateur. La figure~\ref{fig:api_gateway} schématise ces interactions.

\section{Streaming temps réel via SSE}

Le modèle d'exécution du pipeline impose une synchronisation délicate : LangGraph s'exécute de manière synchrone dans un thread, tandis que FastAPI sert les réponses HTTP de manière asynchrone. Le choix de \textit{Server-Sent Events} plutôt qu'un \textit{WebSocket} bidirectionnel se justifie par trois éléments.

\paragraph{Unidirectionnalité du flux.}
Le pipeline pousse vers l'UI (événements de progression) ; les décisions HITL remontent par un endpoint POST distinct (\texttt{/hitl}). Aucun besoin d'un canal bidirectionnel persistant.

\paragraph{Standard HTTP/1.1 sans dépendance.}
SSE est consommé nativement par les navigateurs via \texttt{EventSource} et reste trivial à parser côté serveur Python. Le module \texttt{src/ui/api\_client.py} implémente un parser pur (\texttt{parse\_sse\_chunks}) testable hors du contexte HTTP.

\paragraph{Latence tolérée.}
Les événements (fin de nœud, passage en HITL, erreur) se produisent à l'échelle de la seconde voire de la dizaine de secondes. Un sondage non bloquant à 100\,ms côté serveur suffit, sans surcharger CPU ni bande passante.

L'implémentation utilise une \texttt{queue.Queue} \textit{thread-safe} comme pont entre le \textit{worker} LangGraph et le générateur SSE. Le \textit{worker} pousse des événements typés (\texttt{run\_started}, \texttt{node\_completed}, \texttt{awaiting\_hitl}, \texttt{run\_done}, \texttt{run\_error}) ; le générateur les consomme et les sérialise en JSON. Une sentinelle \texttt{None} marque la fin du flux et ferme proprement la connexion.

\begin{lstlisting}[language=Python,caption={Émission d'un événement typé (\texttt{src/api.py})}]
_push_event(run_id, "run_done", {
    "rapport_chars": len(rapport),
    "candidats": len(candidats),
})
\end{lstlisting}

Le \texttt{StreamingResponse} retourné côté FastAPI ajoute l'en-tête \texttt{X-Accel-Buffering: no}, évitant le \textit{buffering} d'un éventuel \textit{reverse-proxy} en amont.

\section{Human-in-the-loop : signalement par \texttt{threading.Event}}

L'interruption avant le nœud A7 repose sur le mécanisme \texttt{interrupt\_before} de LangGraph. Pour reprendre l'exécution après validation, il faut synchroniser deux univers : le \textit{worker} thread (LangGraph synchrone) et le \textit{handler} HTTP (asynchrone). Le choix de \texttt{threading.Event} plutôt qu'un mécanisme \textit{ad hoc} se justifie par quatre propriétés.

\begin{enumerate}
\item \textbf{Clarté sémantique.} Un \texttt{Event} modélise exactement le problème : un signal d'un thread à un autre. Le code reste lisible : un \texttt{wait()} bloquant côté \textit{worker}, un \texttt{set()} côté \textit{handler}.
\item \textbf{Sécurité.} \texttt{Event.set()} est atomique en CPython ; aucun risque de course entre l'attente et la signalisation.
\item \textbf{Timeout intégré.} \texttt{wait(timeout=600)} libère le \textit{worker} après dix minutes, évitant la fuite de processus en cas d'oubli humain.
\item \textbf{Coût minimal.} Pas de \textit{polling} CPU, pas de file supplémentaire : un seul octet de signalisation par thread.
\end{enumerate}

\begin{lstlisting}[language=Python,caption={Attente bloquante du worker sur la décision HITL (\texttt{src/api.py})}]
hitl_event: threading.Event = _runs[run_id]["_hitl_event"]
got_decision = hitl_event.wait(timeout=_HITL_TIMEOUT)
if not got_decision:
    raise TimeoutError(
        f"Aucune decision HITL recue en {_HITL_TIMEOUT}s..."
    )
decision = _runs[run_id]["_hitl_decision"]
\end{lstlisting}

La séquence complète, schématisée figure~\ref{fig:hitl_event}, est la suivante : (i) le graphe atteint l'\textit{interrupt} et collecte les candidats en attente ; (ii) le \textit{worker} pousse un événement \texttt{awaiting\_hitl} via SSE ; (iii) l'UI affiche un formulaire à trois actions (\textit{approve}, \textit{skip}, \textit{edit}) ; (iv) le recruteur valide via \texttt{POST /hitl} ; (v) l'endpoint stocke la décision et appelle \texttt{event.set()} ; (vi) le \textit{worker} se réveille, applique la décision via \texttt{app\_graph.update\_state()} puis relance le streaming.

\section{Interface graphique NiceGUI}

NiceGUI a été retenu après comparaison avec Streamlit, Gradio, Reflex et un \textit{stack} JavaScript (Next.js, Vite/React). Cinq critères ont été décisifs.

\begin{enumerate}
\item \textbf{Langage unifié.} L'équipe est composée de trois étudiants Python ; imposer React ou Next.js réduirait la capacité de contribution à un seul membre.
\item \textbf{Multi-pages avec navigation.} NiceGUI propose des pages décorées (\texttt{@ui.page}) et un \textit{drawer} latéral (\texttt{src/ui/layout.py}). Streamlit impose une navigation par \texttt{st.selectbox} maladroite ; Gradio ne supporte pas la pagination.
\item \textbf{Asynchrone natif.} NiceGUI s'appuie sur Quasar/Vue.js et permet des \textit{callbacks} \texttt{async}, du SSE et des tâches d'arrière-plan sans \textit{boilerplate}. Streamlit ré-exécute le script entier à chaque interaction, modèle inadapté à un pipeline long avec HITL.
\item \textbf{Composants riches.} Markdown, tables, formulaires, \texttt{ui.mermaid()} pour la visualisation du graphe. Reflex requiert JSX/TypeScript pour le rendu personnalisé.
\item \textbf{Déploiement léger.} Un seul processus Python, pas de serveur Node ; intégration triviale dans le \texttt{docker-compose.yml} existant.
\end{enumerate}

La topologie de l'UI compte cinq pages :
\begin{itemize}[noitemsep]
\item \texttt{/} -- formulaire de lancement, statut de l'API ;
\item \texttt{/runs} -- historique paginé ;
\item \texttt{/runs/\{run\_id\}} -- suivi en direct (Mermaid mis à jour par SSE, \textit{timeline} d'événements, encart HITL) ;
\item \texttt{/rag} -- recherche de profils similaires dans ChromaDB ;
\item \texttt{/metriques} -- tableau d'observabilité.
\end{itemize}

Le composant critique est la consommation du flux SSE. Une coroutine \texttt{consume\_stream()} itère sur les événements, met à jour le diagramme Mermaid et la \textit{timeline} sans bloquer l'interaction utilisateur.

\begin{lstlisting}[language=Python,caption={Consommation SSE asynchrone côté UI (\texttt{src/ui/app.py})}]
async def consume_stream() -> None:
    async for event in stream_events(run_id):
        event_type = event.get("type", "unknown")
        data = event.get("data", {})
        add_event(event_type, data)

        if event_type == "node_completed":
            completed.add(data.get("node", ""))
            diagram.content = _build_mermaid(completed, ...)
            diagram.update()

background_tasks.create(consume_stream())
\end{lstlisting}

Le parser SSE (\texttt{parse\_sse\_chunks}) est volontairement pur : il transforme un bloc de lignes en liste d'événements typés sans dépendance réseau. Cette séparation rend le parser testable indépendamment.

\subsection{Collecte et export de métriques}

Le module \texttt{src/observabilite.py} expose un singleton \texttt{PipelineMetrics}. Chaque nœud invoque \texttt{get\_metrics().fin("nom\_etape", **kwargs)} avec les métriques métier qui le concernent.

\begin{lstlisting}[language=Python,caption={Annotation d'une étape avec ses métriques métier}]
m = get_metrics()
m.fin("A3c_filtre", n_entree=30, n_sortie=8, taux_bruit=0.73)
\end{lstlisting}

À la fin du pipeline, le nœud rapport invoque \texttt{m.exporter()}, qui sérialise les résultats dans \texttt{logs/metriques\_AAAAMMJJ\_HHMMSS.json}. L'historique est consultable via \texttt{GET /metrics} et affiché dans la page \texttt{/metriques} de l'UI.

\subsection{Journalisation structurée}

Le module \texttt{src/logger.py} impose un format ISO 8601 + niveau + composant. Tous les agents et l'API utilisent \texttt{get\_logger("composant")}, ce qui centralise la production sur \texttt{stdout} et la rend capturable par Docker ou \texttt{systemd}.

L'ensemble est livré sous forme d'un \texttt{docker-compose.yml} déclarant trois services : \texttt{ollama} (LLM local), \texttt{sma-api} (FastAPI) et \texttt{sma-ui} (NiceGUI). Deux volumes hôtes (\texttt{./logs} et \texttt{./data}) persistent respectivement les exports d'observabilité et la base ChromaDB. La figure~\ref{fig:docker_compose} en donne la topologie.

\section{Extraits commentés}

\begin{lstlisting}[language=Python,caption={Initialisation d'un \textit{run} et lancement en arrière-plan (\texttt{src/api.py})}]
@app.post("/recruter", response_model=RecruterResponse)
async def recruter(
    req: RecruterRequest, background_tasks: BackgroundTasks
) -> RecruterResponse:
    run_id = str(uuid.uuid4())
    _runs[run_id] = {
        "run_id": run_id,
        "fiche_poste": req.fiche_poste,
        "status": "running",
        "_event_queue": queue.Queue(),       # pont synchrone -> async
        "_hitl_event": threading.Event(),    # signal de decision
        "_with_interrupt": not req.no_interrupt,
    }

    background_tasks.add_task(
        _run_pipeline,
        run_id,
        req.fiche_poste,
        not req.no_interrupt,
    )
    return RecruterResponse(run_id=run_id, status="running", ...)
\end{lstlisting}

La file SSE et l'\textit{Event} HITL sont initialisés dès la requête, ce qui découple le cycle de vie du \textit{run} de celui de la requête HTTP.

\begin{lstlisting}[language=Python,caption={Capture d'événements dans la file SSE (\texttt{src/api.py})}]
def _push_event(run_id: str, event_type: str, data: dict) -> None:
    info = _runs.get(run_id)
    if info is None:
        return
    q = info.get("_event_queue")
    if q is not None:
        q.put({
            "type": event_type,
            "data": {**data, "ts": datetime.now(...).isoformat()},
        })
\end{lstlisting}

Chaque événement reçoit un horodatage ISO et reste sérialisable en JSON. L'absence de \textit{run\_id} en mémoire est tolérée silencieusement, évitant qu'une condition de course ne tue le \textit{worker}.

\begin{lstlisting}[language=Python,caption={Export des métriques en fin de pipeline (\texttt{src/agents/orchestrateur.py})}]
m = get_metrics()
m.noter(
    "pipeline",
    n_profils_bruts=n_bruts,
    n_profils_dedup=n_dedup,
    n_scores=len(candidats_scores),
    n_valides=len(candidats_valides),
)
metriques_export = m.exporter()
metriques_resume = m.resume_texte()
\end{lstlisting}

L'export est non destructif : les métriques restent disponibles en mémoire pour les requêtes ultérieures sur \texttt{/metrics} ou pour la page \texttt{/metriques} de l'UI.
