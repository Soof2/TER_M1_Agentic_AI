\section{Pipeline de sourcing multi-sources (A3)}

Le pipeline de collecte de candidats (agent A3) se décompose en trois sous-agents spécialisés, conformément au patron de séparation des préoccupations. Cette décomposition répond à deux objectifs : (i) clarifier les responsabilités cognitives (\textit{que chercher~?}), logistiques (\textit{comment chercher~?}) et décisionnelles (\textit{quoi conserver~?}) ; (ii) permettre à chaque couche de fonctionner indépendamment et d'être substituée sans répercussion sur les autres.

\paragraph{A3a — Stratège (\texttt{src/agents/chercheur\_stratege.py}).}
Le stratège reçoit le profil de compétences extrait par A2 et produit un plan de recherche structuré. Il invoque le LLM avec un \textit{prompt} imposant quatre familles de requêtes optimisées par source : requêtes web générales (avec opérateurs comme \texttt{intitle:CV}), requêtes LinkedIn ciblées (\texttt{site:linkedin.com/in}), requêtes natives pour l'API GitHub (par exemple \texttt{language:Python location:Paris followers:>5}) et requêtes pour agrégateurs de portfolios (Malt, DoYouBuzz). Le \textit{prompt} impose également le respect strict du niveau d'expérience : pour les niveaux « alternant » ou « stagiaire », des exclusions \texttt{-"senior" -"lead"} sont ajoutées en post-traitement. Le stratège produit un dictionnaire \texttt{requetes\_recherche} avec les clés \texttt{queries\_generales}, \texttt{queries\_linkedin}, \texttt{queries\_github}, \texttt{queries\_cv\_sites} et \texttt{tags\_stackoverflow}. Aucun appel réseau n'est effectué à ce stade : le LLM génère les requêtes de manière déclarative.

\paragraph{A3b — Collecteur (\texttt{src/agents/chercheur\_collecteur.py}).}
Le collecteur exécute en parallèle les requêtes générées par le stratège sur cinq sources gratuites :
\begin{enumerate}
\item DuckDuckGo pour les requêtes web générales, LinkedIn et portfolios ;
\item l'API officielle GitHub Search Users (60 requêtes par heure sans \textit{token}, 5\,000 avec un \textit{token} gratuit) ;
\item l'API Stack Exchange (300 requêtes par jour sans clé, 10\,000 avec clé gratuite).
\end{enumerate}
Chaque source retourne des résultats structurés au format unifié \texttt{\{title, url, body, source\}}. Le collecteur agrège l'ensemble, puis effectue une première déduplication par URL. Aucune décision algorithmique ou linguistique n'est appliquée à ce stade : le nœud reste \textit{stateless} et déterministe. Les journaux reportent le nombre de résultats bruts par source, ce qui facilite l'observabilité du bruit amont.

\paragraph{A3c — Filtre (\texttt{src/agents/chercheur\_filtre.py}).}
Le filtre transforme les hits bruts en profils candidats exploitables via un filtrage algorithmique multi-étapes, sans recours au LLM (voir section suivante). Les profils générés sont limités à \texttt{MAX\_PROFILS\_RECHERCHE} (15 par défaut). Le filtre consigne trois métriques : taux de bruit brut (hits éliminés / hits collectés), nombre d'échecs de \textit{scraping} (repli sur le \textit{snippet} DuckDuckGo) et nombre final de profils exploitables.

La séquence complète suit un flux LangGraph linéaire A3a \(\rightarrow\) A3b \(\rightarrow\) A3c, représenté figure~\ref{fig:pipeline_a3}. Le nœud A3c produit une liste \texttt{profils\_bruts} prête à être consommée par le déduplicateur (A6) puis l'évaluateur (A4).

\section{Filtre anti-bruit algorithmique}

Le filtre d'A3c incarne un choix délibéré : appliquer des règles algorithmiques déterministes plutôt qu'un filtrage par LLM. Trois arguments justifient cette décision.

\paragraph{Coût et latence.}
Un filtrage par LLM supposerait d'invoquer le modèle pour chaque hit brut, amplifiant la latence globale du pipeline (déjà 2 à 5 minutes) et augmentant la charge de calcul. Avec une moyenne de 40 à 50 hits par recherche avant filtrage et 5 à 8 requêtes par source, cela représenterait 200 à 400 appels LLM pour une seule exécution. Les règles algorithmiques (domaines bloqués, mots-clés, fragments d'URL) sont au contraire exécutées en environ une milliseconde par hit.

\paragraph{Déterminisme et auditabilité.}
Les listes \texttt{\_NOISE\_DOMAINS} et \texttt{\_NOISE\_KEYWORDS} sont codifiées en clair, lisibles et modifiables sans réentraînement. Un évaluateur du rapport peut consulter le code source et comprendre exactement quels domaines et termes sont exclus. Un filtre LLM serait au contraire une boîte noire probabiliste : deux exécutions sur les mêmes données pourraient produire des décisions différentes.

\paragraph{Stabilité face au bruit.}
Les domaines d'agrégateurs et les formulations « offre d'emploi » sont stables dans le temps et largement universels. Un LLM, entraîné sur une distribution générale d'Internet, peut être confus par des cas limites (un profil LinkedIn dont la description évoque le mot « candidature ») ou dévier si la température est élevée. Les règles algorithmiques restent insensibles à ces variations.

\paragraph{Limitations acceptées.}
Le filtre ne détecte pas le bruit sémantique fin (CV générique, profil copié d'un agrégateur). Ce résidu est absorbé en aval par l'évaluateur A4, qui s'appuie sur la similarité sémantique via le RAG et le jugement du LLM. Cet arbitrage entre filtrage précoce et filtrage tardif est documenté dans les journaux : un taux de suppression de 40 à 60\,\% des hits bruts est observé en pratique.

\section{Mémoire RAG calibrée par fiche de poste}

La mémoire vectorielle (\texttt{src/tools/rag.py}, classe \texttt{MemoireRAG}) résout un problème structurel du RAG naïf inter-fiches. Dans un système de recrutement multi-runs, chaque candidat reçoit un score d'évaluation dans le contexte d'une fiche de poste donnée. Réutiliser ce score comme signal de calibration pour une autre fiche est trompeur : un candidat évalué 85/100 pour « Développeur Python senior » n'est pas pertinent comme référence pour « Chef de projet marketing », même en cas de similarité textuelle superficielle.

\subsection{Architecture à deux collections}

\subsection{Procédure de lecture en deux temps}

Lorsque A4 évalue un nouveau candidat pour une fiche courante, l'agent appelle \texttt{rechercher\_similaires(profil\_brut, fiche\_poste=...)}. Le code effectue alors :
\begin{enumerate}
\item un \textit{embedding} de la fiche courante via \texttt{SentenceTransformer all-MiniLM-L6-v2} ;
\item une interrogation de \texttt{fiches\_poste} avec un seuil de similarité cosinus $\geq 0{,}5$ ;
\item si aucune fiche stockée ne dépasse le seuil, la fonction retourne une liste vide \textit{sans injecter de contexte RAG} ;
\item sinon, une interrogation de \texttt{candidats\_evalues} avec le filtre \texttt{where=\{"fiche\_id": \{"\$in": [...]\}\}}, qui ne remonte que les candidats rattachés à des fiches similaires.
\end{enumerate}

Cette procédure élimine le biais inter-fiches tout en maintenant un silence explicite (aucun contexte injecté) sur une fiche inédite. Au premier \textit{run}, le RAG ne produit rien ; sur le second \textit{run} avec une fiche analogue, les candidats du premier deviennent une référence de calibration valide.

\subsection{Garanties testées}

Trois invariants sont vérifiés unitairement (voir \texttt{tests/test\_rag.py}, classe \texttt{TestCalibrationParFiche}) :
\begin{enumerate}
\item \textbf{Hash stable} : deux textes identiques aux espaces et à la casse près produisent le même \texttt{fiche\_id}.
\item \textbf{Isolation inter-fiches} : si deux fiches très différentes coexistent en base et que la fiche courante ressemble à la première, seuls les candidats de la première remontent.
\item \textbf{Repli silencieux} : si aucune fiche stockée ne dépasse le seuil, la fonction retourne \texttt{[]} sans exception.
\end{enumerate}

Le seuil de 0,5 est un compromis : trop bas, le RAG retombe dans le biais inter-fiches ; trop haut, il ne se déclenche presque jamais. Le seuil est exposé en paramètre, donc ajustable empiriquement.

\section{Justification des choix techniques}

\subsection{Sources gratuites uniquement (DuckDuckGo, GitHub, Stack Overflow)}

\paragraph{Besoin.} Sourcer des candidats sans coût d'exploitation, dans un cadre TER sans budget commercial.

\paragraph{Alternatives.} L'API LinkedIn officielle requiert une approbation peu accessible et ses conditions d'utilisation prohibent le recrutement automatisé : \textit{ban risk} et exposition juridique. Malt impose un contrat commercial pour l'accès programmatique. Indeed et Glassdoor sont fermés aux tiers ; un \textit{scraping} entraînerait une violation des CGU et des risques RGPD significatifs.

\paragraph{Décision.} DuckDuckGo (via la bibliothèque \texttt{duckduckgo-search}, libre et sans authentification), GitHub Search Users API (publique, gratuite, dédiée aux profils) et Stack Exchange API (publique, orientée tags techniques). Ces trois sources couvrent une part substantielle des profils en ligne sans exposition juridique ni coût.

\paragraph{Tradeoffs.} DuckDuckGo n'offre pas les filtres avancés de LinkedIn et remonte du bruit d'agrégateurs (atténué par A3c). GitHub est puissant pour les profils techniques mais limité aux développeurs ayant un compte public. Stack Overflow privilégie les développeurs expérimentés. La complémentarité des trois canaux absorbe ces limitations.

\subsection{Filtrage algorithmique plutôt que LLM}

Voir section précédente. Argumentaire synthétique : coût linéaire sans appel LLM, déterminisme et auditabilité, stabilité face aux variations.

\subsection{Décomposition A3 en trois sous-agents}

\paragraph{Besoin.} Clarifier les responsabilités du \textit{sourcing} pour faciliter le débogage, la substitution et la compréhension architecturale.

\paragraph{Alternatives.} Un agent A3 monolithique, plus simple à implémenter mais mêlant cognition, logistique et décision ; ou trois scripts isolés en dehors du graphe LangGraph, qui perdraient les \textit{checkpoints} et l'observabilité unifiée.

\paragraph{Décision.} Trois nœuds LangGraph distincts formant un pipeline linéaire. Chaque nœud a une entrée et une sortie clairement définies, un \textit{logger} dédié et des métriques reportées à \texttt{get\_metrics()}.

\paragraph{Tradeoffs.} Trois passages dans le \textit{runtime} LangGraph ; surcoût négligeable comparé aux appels réseau et au \textit{scraping}.

\subsection{\textit{Embeddings} locaux (SentenceTransformers \texttt{all-MiniLM-L6-v2})}

\paragraph{Besoin.} Produire des \textit{embeddings} cohérents, déterministes et disponibles hors ligne.

\paragraph{Alternatives.} \texttt{text-embedding-3-large} d'OpenAI (qualité supérieure mais clé API, coût et exposition réseau) ; Hugging Face Inference API (gratuité limitée).

\paragraph{Décision.} \texttt{SentenceTransformer all-MiniLM-L6-v2} (384 dimensions, 80\,Mo, latence d'environ 50\,ms par \textit{embedding} sur CPU). Avantages : déploiement Docker indépendant du réseau, coût marginal nul, déterminisme, taille adaptée aux recherches cosinus dans ChromaDB.

\paragraph{Tradeoffs.} Qualité sémantique légèrement inférieure aux modèles à 3\,072 dimensions ; suffisante pour discriminer des fiches au seuil 0,5. La couche \texttt{MemoireRAG} isole le modèle, qui peut être substitué sans modification des appelants.

\subsection{ChromaDB local plutôt que service externe}

\paragraph{Besoin.} Stocker et interroger des \textit{embeddings} avec filtrage par métadonnées (\texttt{fiche\_id}).

\paragraph{Alternatives.} Pinecone (cloud, dépendance réseau, quota gratuit limité) ; Weaviate (puissant mais plus lourd) ; Qdrant (très bon, mais surcoût opérationnel).

\paragraph{Décision.} ChromaDB en client persistant sur disque (\texttt{./data/chromadb}). Justifications : déploiement Docker cohérent, données locales, latence négligeable, simplicité pour un prototype TER.

\paragraph{Tradeoffs.} ChromaDB n'est pas optimisé pour les très grandes échelles (10\,M+ vecteurs) ; non bloquant à notre échelle. La couche \texttt{MemoireRAG} encapsule les appels, rendant une migration ultérieure straightforward.

\subsection{RAG calibré plutôt que RAG global naïf}

\paragraph{Besoin.} Améliorer la calibration des évaluations en s'appuyant sur des candidats évalués précédemment, sans introduire de biais inter-fiches.

\paragraph{Alternatives.} RAG global (biais inter-fiches structurel) ; \textit{fine-tuning} du LLM (coût et risque d'\textit{overfitting} disproportionnés à un TER) ; index manuel \textit{few-shot} par fiche (fragile).

\paragraph{Décision.} RAG calibré par fiche (deux collections, filtrage par \texttt{fiche\_id}). Élimine le biais structurel sans surcoût algorithmique notable, garantit un silence explicite sur les fiches inédites, propriétés testables.

\paragraph{Tradeoffs.} Apport nul au premier \textit{run} ; propriété recherchée plutôt que défaut.

\begin{lstlisting}[language=Python,caption={Hash déterministe de la fiche de poste (\texttt{src/tools/rag.py})}]
def hash_fiche(texte: str) -> str:
    """Hash stable -- meme texte au whitespace pres = meme id."""
    normalise = " ".join(texte.split()).lower()
    return hashlib.sha1(normalise.encode("utf-8")).hexdigest()[:12]
\end{lstlisting}

Ce hash garantit l'idempotence : appeler deux fois \texttt{ajouter\_fiche\_poste()} avec le même texte ne crée pas de doublon. La normalisation rend le hash robuste aux variations de formatage.

\begin{lstlisting}[language=Python,caption={Recherche RAG calibrée par fiche (\texttt{src/tools/rag.py})}]
def _fiches_similaires_ids(
    self, fiche_texte: str, seuil: float, n_max: int = 3
) -> list[str]:
    """Retourne les fiche_ids au-dessus du seuil."""
    if self._fiches.count() == 0:
        return []
    results = self._fiches.query(
        query_texts=[fiche_texte[:3000]],
        n_results=min(n_max, self._fiches.count()),
        include=["distances"],
    )
    ids = results.get("ids", [[]])[0]
    distances = results.get("distances", [[]])[0]
    retenus = [
        fid for fid, dist in zip(ids, distances)
        if (1 - dist) >= seuil
    ]
    return retenus

def rechercher_similaires(
    self,
    profil_brut: str,
    fiche_poste: str | None = None,
    n_results: int = 3,
    seuil_fiche: float = _SEUIL_FICHE_DEFAUT,
) -> list[dict]:
    """Recherche calibree si fiche_poste est fourni."""
    if fiche_poste is not None:
        fiche_ids = self._fiches_similaires_ids(fiche_poste, seuil_fiche)
        if not fiche_ids:
            return []   # silence si aucune fiche comparable
        where_clause = {"fiche_id": {"$in": fiche_ids}}

    results = self._candidats.query(
        query_texts=[profil_brut[:2000]],
        n_results=n_results,
        include=["metadatas", "distances"],
        where=where_clause if fiche_poste else None,
    )
\end{lstlisting}

Cette implémentation illustre la logique en deux phases : d'abord filtrer les fiches par similarité seuillée, ensuite restreindre la recherche de candidats aux fiches retenues. Si aucune fiche n'atteint le seuil, la fonction retourne immédiatement une liste vide, écartant le biais inter-fiches.

def stratege_node(state: GraphState) -> dict:

Le stratège combine la générativité du LLM (produire des requêtes adaptées) avec un filtrage \textit{post hoc} pour garantir le respect du niveau d'expérience. Cette hybridation équilibre flexibilité et robustesse.

La déduplication par URL est essentielle : DuckDuckGo et l'API GitHub peuvent retourner le même profil. L'étape précède le \textit{scraping}, ce qui évite des appels réseau redondants. Les métriques journalisées rendent compte du taux de redondance inter-sources.
